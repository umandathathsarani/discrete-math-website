Lecture 10: Analysis of Algorithms

1. Introduction to Algorithms

An Algorithm is a step-by-step procedure for solving a problem, much like a recipe or set of instructions (e.g., sorting lists, searching for items, calculating values).

Why Analyze Algorithms?

  • To predict performance before actual implementation.
  • To compare different approaches and choose the right tool for the job.
  • To identify bottlenecks and opportunities for improvement.

Key Questions & How We Measure

We ask: How long will this algorithm take (Time complexity)? How much memory will it need (Space complexity)? Does it scale well?

Crucial Note: We do NOT measure with actual time in seconds. Actual time depends heavily on hardware, programming languages, and specific implementations. Instead, we count basic operations (comparisons, arithmetic operations, assignments) and focus on how these grow with the input size.

2. Why We Need Asymptotic Notation

Asymptotic notation provides a mathematical framework for analyzing algorithms due to several fundamental reasons:

  • Focus on Scalability & Hardware Independence: As input sizes increase dramatically, true efficiency differences emerge. Asymptotic notation abstracts away hardware-specific factors, allowing fair, universal comparisons.
  • Simplifies Analysis: It ignores constant factors and eliminates lower-order terms. It concentrates on the dominant factor determining long-term performance. (Example: 3n2 + 20n + 500 simplifies to Θ(n2) because n2 dominates as n grows large).
  • Theoretical Foundation: Creates a standardized vocabulary and mathematical rigor, preventing us from relying solely on empirical testing.

3. Asymptotic Notations

These notations describe the behavior of functions in the limit (as input sizes grow towards infinity), allowing us to compare the "sizes" of functions.

  • O (Big O): Means "less than or equal to" (≤). Gives an asymptotic upper bound. Example: If an algorithm is O(n2), it won't grow faster than n2.
  • Ω (Big Omega): Means "greater than or equal to" (≥). Gives an asymptotic lower bound. Example: If an algorithm is Ω(n), it grows at least as fast as n.
  • Θ (Big Theta): Means "equal to" (=). Gives a tight bound. Example: If an algorithm is Θ(n log n), it grows exactly at the rate of n log n.
  • o (Little o): Means "strictly less than" (<). Example: o(n2) grows strictly slower than n2.

4. Formalizing Big O-Notation

Introduced by Paul Bachmann in 1892, we use Big O-notation to give an upper bound on a function.

Definition: O(g(n)) = { f(n) : there exist positive constants c and n0 such that 0 ≤ f(n) ≤ c · g(n) for all n ≥ n0 }

If f(n) is in O(g(n)), we write f(n) = O(g(n)).

Figure 2: After point n0, the function f(n) is always bounded above by c·g(n).

Polynomial Theorem

In general: anxn + ... + a1x + a0 is O(xn) for any real numbers and any non-negative n.

Examples of Big O Calculation

  • T(n) = 3 + 5n + 3n2 → O(n2)
  • f(n) = 2n + n2 + 8n + 7 → O(2n)
  • T(n) = n + log n + 6 → O(n)

Code Fragment Example

for i ← 1 to n
    for j ← 1 to i
        Print j

This fragment features a nested loop where the inner loop depends on the outer loop. The total executions sum to n(n+1)/2, which yields O(n2).

5. Recurrence Relations

A recurrence relation is a mathematical function that defines the running time of a recursive function in terms of its value on smaller inputs.

Example: Factorial Function

int factorial(int n) {
    if (n == 0) return 1; // Base case (Time c)
    else return (n * factorial(n-1)); // Recursive case (Time e + T(n-1))
}

The total time T(n) = c + e + T(n-1). Solving this via the iteration method (repeated substitution) yields a linear time complexity.

6. Methods for Solving Recurrences

There are three primary methods used to solve recurrences mathematically:

  • 1. The Substitution Method
    Guess the exact form of the solution, then use mathematical induction to find the constants and prove that the bound holds. For example, to prove T(n) = 2T(n/2) + n is O(n log n), we guess T(n) ≤ c·n log n and substitute it back into the recurrence to verify.
  • 2. The Recursion-Tree Method
    Draw out a tree where each node represents the cost of a single subproblem. Sum the costs across each level, and then sum all levels to determine the total cost. This is highly useful for generating a good "guess" for the Substitution method.

Figure 3: Recursion tree for T(n) = 3T(n/4) + cn2. The total cost resolves to O(n2).

7. The Master Method / Master Theorem

The Master Method applies specifically to recurrences of the form:

T(n) = aT(n/b) + f(n)

Where a ≥ 1 and b > 1 are constants, and f(n) is an asymptotically positive function. It describes dividing a problem into a subproblems, each of size n/b, with f(n) representing the cost to divide and combine.

To use the theorem, compare nlogba against f(n):

  • Case 1: If f(n) is polynomially smaller than nlogba (f(n) = O(nlogba - ε)).
    Solution: T(n) = Θ(nlogba)
  • Case 2: If f(n) is roughly equal to nlogba (f(n) = Θ(nlogba logk n)).
    Solution: T(n) = Θ(nlogba logk+1 n)
  • Case 3: If f(n) is polynomially greater than nlogba (f(n) = Ω(nlogba + ε)), and a regularity condition holds.
    Solution: T(n) = Θ(f(n))

Master Theorem Examples

  • Example Case 1: T(n) = 9T(n/3) + n.
    a = 9, b = 3, f(n) = n. Calculate nlog39 = n2.
    Since f(n) = n is polynomially smaller than n2, this is Case 1.
    Result: T(n) = Θ(n2).
  • Example Case 2: T(n) = T(2n/3) + 1.
    a = 1, b = 3/2, f(n) = 1. Calculate nlog3/21 = n0 = 1.
    Since f(n) = 1 is equal to n0, this is Case 2 (with k=0).
    Result: T(n) = Θ(log n).
  • Example Case 3: T(n) = 3T(n/4) + n log n.
    a = 3, b = 4, f(n) = n log n. Calculate nlog43 ≈ n0.793.
    Since f(n) = n log n is polynomially strictly greater than n0.793, this is Case 3.
    Result: T(n) = Θ(n log n).
